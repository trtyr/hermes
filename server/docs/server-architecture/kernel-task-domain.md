# 任务流转内核

这篇文档描述 `server` 中负责任务流转的能力域。

这里讨论的是：

- 单个 Agent 任务派发
- 广播任务
- 任务取消
- 任务状态更新

这里不讨论：

- 前端列表分页
- Agent 协议帧的原始格式

## 1. 对应代码

- `src/kernel/runtime/task_flow.rs`
- `src/kernel/service/tasks/commands.rs`
- `src/kernel/service/tasks/queries.rs`
- `src/kernel/service/tasks/mod.rs`
- `src/kernel/state/task_state.rs`
- `src/kernel/storage/tasks.rs`
- `src/api/tasks/*`

## 2. 任务域的核心问题

这个能力域解决的问题是：

“一个操作请求，如何变成后端可追踪、可持久化、可回报的任务。”

它关心的不只是“发出命令”，还包括：

- 任务创建
- 目标选择
- 派发状态
- 结果状态
- 广播父子任务关系

## 3. 当前入口

任务域当前通过 `TaskKernelMessage` 进入运行时。

主要入口包括：

- `Dispatch`
- `Broadcast`
- `Cancel`
- `DisconnectAgent`

这说明任务域在当前代码里已经是独立消息域，而不是 Agent handler 的附属逻辑。

## 4. 当前流转

### 4.1 单播任务

1. 外层创建任务请求
2. 任务域在内存里创建任务记录
3. 若目标 Agent 在线，则向该 session 发送 `DispatchTask`
4. 更新任务状态并发布事件
5. 若目标不在线，则把任务标记为失败

### 4.2 广播任务

1. 创建父任务
2. 按在线 Agent 生成子任务
3. 分别派发到各 Agent
4. 父子任务共同进入持久化和事件流

### 4.3 取消任务

1. 递归取消子任务
2. 若目标仍在线，则向 Agent 发送取消指令
3. 更新取消状态并发布事件

## 5. 任务域为什么要独立

如果任务派发散落在 Agent 生命周期或 API 层，通常会出现这些问题：

- 任务状态和派发状态耦合不清
- 广播任务没有统一父子关系模型
- 取消逻辑四处分散

把任务流转单独做成能力域之后，任务就有了自己完整的生命周期。

## 6. 状态与持久化

任务域依赖两层：

- `state/task_state`
  维护任务的运行态和父子关系
- `storage/tasks`
  维护持久化记录和查询

这意味着：

- 任务的“当前状态”由内核掌握
- 任务的“历史事实”由 SQLite 记住

## 7. 设计规则

关于任务域，建议固定遵守下面几条：

1. 任务创建和状态流转必须进入任务域
2. 广播任务必须显式建模父子关系
3. “未在线导致失败”属于任务域语义，不要散在 API 层
4. 任务取消必须由任务域统一处理

## 8. 当前评价

`task_flow.rs` 目前已经成为比较明确的任务内核。

后续如果要继续优化，重点不是再加分支，而是继续把任务状态规则压回 `state/task_state`。
