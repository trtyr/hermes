# 控制通道设计

这份文档定义 `server <-> agent` 之间这条控制通道的设计语义。

它强调的是“控制平面怎么设计更清晰、更稳”，而不是隐蔽性或攻击性。

## 1. 范围边界

系统中有两条不同的平面：

- `frontend <-> backend`
  管理面
- `backend <-> agent`
  控制面

控制面服务于：

- listener 接入
- Agent 注册
- beacon 保活
- 任务派发
- 任务结果回传
- 命令会话排队
- 运行时配置更新

## 2. 当前控制通道模型

当前模型具有这些特征：

- 长连接 TCP
- 一行一个 JSON 帧
- Agent 主动 beacon
- 后端 watchdog 判超时
- 后端维护命令会话队列

## 3. 为什么说它是“控制通道”

因为这条链路承载的是：

- 身份建立
- 存活维持
- 指令下发
- 结果回传
- 运行时配置同步

## 4. 企业内控取向

当前设计明确排除：

- 隐蔽传输
- 混淆协议
- 规避分析
- 攻击性时间策略

保留的是：

- 明确握手
- 明确认证
- 明确能力协商
- 明确任务和会话模型

## 5. 认证模式

当前后端支持：

- 未配置 token
- `plain_token`
- `challenge_response`

推荐生产模式：

- 配置 `auth.agent_token`
- 使用 `auth.agent_auth_mode = "challenge_response"`

## 6. 为什么挑战应答有价值

相较于明文 token 直接进 `register`，挑战应答的好处是：

- 避免每次注册都重复发原始 token
- 让证明绑定到本次连接的随机 nonce
- 在握手阶段显式暴露协议版本和 transport profile
- 为未来能力协商保留标准入口
