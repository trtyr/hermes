# 第5章 系统实现

系统实现阶段将第4章的设计方案转化为可运行的代码。Hermes平台由三个独立子项目构成：服务端（Rust）、受控端Agent（Rust）和Web控制台（Vue 3）。服务端采用微内核架构，所有业务逻辑通过消息调度驱动；Agent以同步阻塞模型运行，追求二进制体积最小化和运行时依赖归零；控制台基于组件化开发，通过HTTP API和WebSocket与服务端交互。本章按开发环境、服务端、控制台、Agent的顺序逐层展开，重点描述关键模块的实现思路和核心代码。

## 5.1 系统开发环境与工具链

整个项目在一台macOS工作站上完成开发。服务端和Agent共享Rust工具链，版本为1.85.0，编译器edition设置为2024（服务端）和2021（Agent），两者差异源于Agent需要兼容更广泛的Rust版本以简化交叉编译流程。构建系统使用Cargo，没有引入workspace级别的monorepo管理工具，三个子项目各自拥有独立的Cargo.toml或package.json。

服务端的依赖栈以tokio异步运行时为核心，配合axum 0.8提供HTTP和WebSocket服务，rusqlite处理SQLite持久化。Agent侧仅使用标准库和少量第三方库（serde、base64、encoding_rs），不依赖任何异步运行时，所有操作通过标准库的std::thread和std::sync原语完成。Agent的Release编译配置了opt-level="z"、codegen-units=1、LTO=true、panic="abort"和strip=true，将最终二进制控制在最小体积。

Web控制台基于Node.js 20 LTS，构建工具为Vite 6，UI框架选用Ant Design Vue 4，状态管理使用Pinia，终端模拟使用xterm.js。表5-1汇总了开发环境的完整配置。

## 5.2 服务端核心实现

### 5.2.1 微内核运行时与消息调度

服务端的微内核运行时是整个控制平面的中枢。其核心思路是把所有业务操作抽象为KernelMessage，由一条集中的消息循环驱动状态变更和副作用产生。这种设计使得状态修改只发生在kernel/state层，其他任何模块都不直接操作共享状态。

KernelMessage是一个枚举类型，按领域划分为四个子类：Agent消息处理连接与注册生命周期，Task消息处理任务分发与广播，CommandSession消息处理交互式终端会话，Proxy消息处理代理转发。其定义如下：

```rust
#[derive(Debug)]
pub enum KernelMessage {
    Agent(AgentKernelMessage),
    Task(TaskKernelMessage),
    CommandSession(CommandSessionKernelMessage),
    Proxy(ProxyKernelMessage),
}
```

以Agent领域为例，AgentKernelMessage包含Connected、Disconnected、Frame、UpdateBeaconConfig、SweepHeartbeats五个变体。其中Frame变体携带Agent上报的原始消息帧，由运行时分发到对应的处理函数：

```rust
#[derive(Debug)]
pub enum AgentKernelMessage {
    Connected {
        session_id: u64,
        listener_id: Option<i64>,
        listener_name: Option<String>,
        peer_addr: SocketAddr,
        sender: mpsc::UnboundedSender<ServerCommand>,
    },
    Disconnected { session_id: u64 },
    Frame { session_id: u64, frame: AgentMessage },
    UpdateBeaconConfig { /* ... */ },
    SweepHeartbeats,
}
```

消息的传递通道由KernelBus提供，它是对tokio::sync::mpsc的薄封装：

```rust
#[derive(Clone)]
pub struct KernelBus {
    message_sender: mpsc::Sender<KernelMessage>,
}
```

内核状态存储在KernelState中，用Arc<RwLock<KernelState>>包裹以支持并发读写。KernelState内部维护了sessions、agent_index、tasks、command_sessions、command_executions、proxy_sessions、proxy_streams等多个HashMap，以及一系列pending HashMap用于追踪异步操作的回调：

```rust
pub struct KernelState {
    pub(super) sessions: HashMap<u64, AgentSession>,
    pub(super) agent_index: HashMap<String, u64>,
    pub(super) tasks: HashMap<String, TaskRecord>,
    pub(super) command_sessions: HashMap<String, CommandSessionRecord>,
    pub(super) command_executions: HashMap<String, CommandExecutionRecord>,
    pub(super) proxy_sessions: HashMap<String, ProxySessionRecord>,
    pub(super) proxy_streams: HashMap<String, ProxyStreamRecord>,
    // ... pending maps omitted
}
```

运行时的消息循环在kernel_loop函数中运转。它从KernelBus的接收端持续拉取消息，根据消息类型路由到对应的领域处理函数。副作用（持久化、事件广播）通过RuntimePorts结构统一发出，而不是让处理函数直接操作存储层或事件总线：

```rust
pub(super) async fn kernel_loop(
    mut receiver: mpsc::Receiver<KernelMessage>,
    state: Arc<RwLock<KernelState>>,
    events: EventBus,
    storage: Storage,
) {
    let effects = RuntimePorts::new(events, storage);
    while let Some(message) = receiver.recv().await {
        match message {
            KernelMessage::Agent(msg) => {
                route_agent_message(&state, &effects, msg).await;
            }
            KernelMessage::Task(msg) => {
                route_task_message(&state, &effects, msg).await;
            }
            KernelMessage::CommandSession(msg) => {
                route_command_session_message(&state, &effects, msg).await;
            }
            KernelMessage::Proxy(msg) => {
                route_proxy_message(&state, &effects, msg).await;
            }
        }
    }
}
```

RuntimePorts封装了两个端口：EventPublisher负责将状态变更事件广播到WebSocket，StatePersistence负责将变更异步写入SQLite。通过这种端口抽象，运行时处理函数不需要了解底层存储和推送的具体实现，只调用effects.publish()或effects.task_updated()即可。

对外暴露的统一入口是KernelHandle，它持有bus、state、events、storage、auth以及一组AtomicU64计数器的引用。API层和网关层不直接操作内核状态，而是通过KernelHandle上挂载的各个Facade（如TaskFacade、CommandSessionFacade）发起请求。Facade内部将业务意图编码为KernelMessage发送到内核总线，必要时通过oneshot channel等待内核的响应。这套机制保证了数据流的单向性：外部输入永远经过Facade→消息→状态变更的路径，不存在反向依赖。

### 5.2.2 HTTP API路由实现

服务端的HTTP API基于axum框架构建，所有路由在api/mod.rs中集中注册。run_http_api函数接收一个KernelHandle实例，构建Router并绑定到指定地址。路由注册涵盖了认证、Agent管理、任务分发、交互式终端、监听器管理、Agent构建、代理转发、审计日志、仪表盘和WebSocket事件等全部功能：

```rust
let app = Router::new()
    .route("/health", get(system::health))
    .route("/auth/login", post(auth::login))
    .route("/agents", get(agents::list_agents))
    .route("/agents/{agent_id}/tasks", post(agents::dispatch_task))
    .route("/agents/{agent_id}/upload", post(agents::upload_file))
    .route("/agents/{agent_id}/download", post(agents::download_file))
    .route("/agents/{agent_id}/browse", post(agents::browse_file))
    .route("/command-sessions/{session_id}/execute",
           post(command_sessions::execute_command_session))
    .route("/command-sessions/{session_id}/close",
           post(command_sessions::close_command_session))
    .route("/events/ws", get(system::ws_events))
    .route("/web/terminal/ws", get(web_terminal::terminal_ws_events))
    // ... more routes
    .layer(middleware::from_fn(log_http_request))
    .layer(cors)
    .with_state(AppState { kernel });
```

每个API handler遵循统一的工作模式：先通过authorize_api函数校验请求身份，再调用KernelHandle上对应的Facade方法执行业务逻辑，返回JSON响应。authorize_api支持三种认证方式共存——Web会话Cookie、x-api-token请求头、以及Agent令牌。中间件层使用tower_http的CorsLayer处理跨域，log_http_request中间件记录每条请求的方法、路径、状态码和耗时。

以Agent列表查询为例，handler函数只需要几行代码，因为查询逻辑全部封装在Facade中：

```rust
// API handler: list agents
pub async fn list_agents(
    headers: HeaderMap,
    State(state): State<AppState>,
) -> impl IntoResponse {
    if let Some(response) = authorize_api(&state, &headers, None) {
        return response;
    }
    let agents = state.kernel.agent_queries().snapshots().await;
    Json(agents)
}
```

这种分层设计使得API层只负责传输协议适配，不包含任何业务判断。当需要新增功能时，只需要在Router上注册新路由，handler中调用对应的Facade方法即可，不需要触碰内核层的代码。

### 5.2.3 WebSocket实时推送实现

Web控制台需要实时感知Agent的上下线、任务执行结果、终端输出等事件。服务端通过/events/ws端点提供WebSocket长连接，客户端订阅后即可接收所有状态变更的推送。

WebSocket handler在连接建立后做了两件事：立即发送当前所有在线Agent的快照（Snapshot事件），然后进入事件循环，持续从EventBus的broadcast channel中读取新事件并推送给客户端：

```rust
async fn handle_ws(mut socket: WebSocket, state: AppState) {
    // Send initial snapshot
    let snapshot = WebEvent::Snapshot {
        agents: state.kernel.agent_queries().snapshots().await,
    };
    if send_ws_event(&mut socket, &snapshot).await.is_err() {
        return;
    }

    let mut event_rx = state.kernel.events().subscribe();

    loop {
        tokio::select! {
            event = event_rx.recv() => {
                match event {
                    Ok(payload) => {
                        if socket.send(Message::Text(payload.into())).await.is_err() {
                            break;
                        }
                    }
                    Err(RecvError::Lagged(_)) => continue,
                    Err(RecvError::Closed) => break,
                }
            }
            inbound = socket.recv() => {
                match inbound {
                    Some(Ok(Message::Close(_))) | None => break,
                    _ => {}
                }
            }
        }
    }
}
```

EventBus基于tokio::sync::broadcast实现。内核运行时在处理完每条消息后，通过RuntimePorts.publish()将WebEvent发送到EventBus。broadcast channel的容量设置了上限，当消费速度跟不上生产速度时会产生Lagged错误，此时选择丢弃旧事件并继续，不会阻塞内核的消息循环。

WebEvent是一个枚举类型，覆盖了Agent连接、注册、心跳、断开、更新、删除，以及任务分发、结果返回、Agent构建状态变更等所有前端需要感知的事件。控制台的Pinia store在收到这些事件后更新本地状态，Vue的响应式系统自动触发UI重渲染。

### 5.2.4 Agent注册与会话管理实现

Agent通过TCP连接到服务端的监听器端口后，整个注册流程由内核的agent_lifecycle模块驱动。当监听器检测到新TCP连接时，生成一个session_id并通过KernelMessage::Agent::Connected消息通知内核。内核在KernelState的sessions HashMap中创建一条AgentSession记录，该记录初始时没有agent_id，仅包含session_id、peer_addr和发送通道。

Agent在建立连接后会立即发送Register消息，携带自身的主机名、用户名、操作系统、架构、PID、内网IP、心跳间隔、抖动系数和认证令牌等字段。Register消息使用serde的tag=type属性进行JSON序列化，其定义如下：

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(tag = "type", rename_all = "snake_case")]
pub enum AgentMessage {
    Register {
        agent_id: String,
        hostname: String,
        username: Option<String>,
        os: Option<String>,
        arch: Option<String>,
        pid: Option<u32>,
        internal_ip: Option<String>,
        sleep_interval: u64,
        jitter: u32,
        token: Option<String>,
        privilege: String,
        // ... other fields
    },
    Heartbeat { agent_id: Option<String> },
    TaskResult { task_id: String, success: bool, output: String },
    // ... other variants
}
```

内核收到Register消息后，执行以下处理：校验Agent令牌（如果配置了agent_token），将AgentSession中的字段补全，在agent_index中建立agent_id到session_id的映射，通过RuntimePorts持久化Agent上线记录并广播agent_registered事件到所有WebSocket客户端。

心跳机制通过Heartbeat消息维持。Agent按配置的sleep_interval定时发送Heartbeat，内核收到后更新AgentSession的last_seen时间戳并广播agent_heartbeat事件。内核还运行了一个定时SweepHeartbeats任务，当某个Agent超过阈值时间没有心跳时，将其标记为离线。

### 5.2.5 交互式终端会话实现

交互式终端是Hermes最具技术复杂度的功能之一。操作员在Web界面输入命令，命令经服务端转发到Agent执行，执行产生的stdout/stderr实时回传到Web终端显示。整个数据流经过三级跳：浏览器→服务端→Agent→服务端→浏览器。

服务端侧的实现分为HTTP API和WebSocket两个通道。HTTP负责会话管理（打开、关闭、提交命令），WebSocket负责实时推送执行结果。open_terminal_session handler接收agent_id参数，校验Agent在线状态后调用CommandSessionFacade.open()方法，该方法通过oneshot channel向内核发送OpenCommandSession消息并等待结果：

```rust
pub(crate) async fn open_terminal_session(
    headers: HeaderMap,
    State(state): State<AppState>,
    Json(request): Json<WebTerminalOpenRequest>,
) -> impl IntoResponse {
    // Auth check
    if let Some(response) = authorize_api(&state, &headers, None) {
        return response;
    }
    // Verify agent is online
    if !state.kernel.agent_queries().is_connected(&request.agent_id).await {
        return (StatusCode::CONFLICT, Json(/* error */)).into_response();
    }
    // Open session via facade
    match state.kernel.command_sessions()
        .open(request.agent_id.clone(), operator).await
    {
        Ok(session) => (StatusCode::CREATED, Json(/* session data */)).into_response(),
        Err(error) => web_terminal_error_response(error),
    }
}
```

命令执行采用queue模式。操作员提交命令后，服务端将命令加入CommandSessionRecord的queued_command_ids队列，通过KernelMessage::CommandSession::Queue消息发送到内核，内核再将ServerCommand::ExecuteCommandSession下发给Agent。Agent执行完毕后回传CommandSessionResult消息，内核更新状态并通过EventBus广播。Web终端的WebSocket连接订阅了EventBus，收到对应session的事件后解析stdout/stderr/cwd/exit_code，通过xterm.js渲染到终端界面。

内核为每个CommandSession维护了一个CommandSessionRecord，记录会话ID、所属Agent、当前工作目录、状态（open/closed）、活跃命令ID和排队命令队列：

```rust
#[derive(Clone)]
pub struct CommandSessionRecord {
    pub command_session_id: String,
    pub agent_id: String,
    pub cwd: String,
    pub status: CommandSessionStatus,
    pub created_by: String,
    pub created_at: u64,
    pub last_active_at: u64,
    pub active_command_id: Option<String>,
    pub queued_command_ids: VecDeque<String>,
}
```

CommandExecutionRecord则追踪每条命令的完整生命周期，从排队、分发、开始执行到结束，每个阶段都有独立的时间戳，支持精确的命令耗时统计和超时检测。

### 5.2.6 Agent二进制构建服务实现

服务端提供了Agent二进制的在线构建能力。操作员在Web界面指定目标平台（如x86_64-pc-windows-msvc）、监听器、服务器地址、心跳参数和编译profile后，服务端异步执行cargo build生成定制化的Agent可执行文件。

构建服务由AgentBuildFacade提供入口。build_agent_binary方法首先解析构建参数，确定目标三元组、服务器地址（默认从监听器绑定地址推导）和心跳配置，在数据库中创建一条AgentBuildRecord，然后通过tokio::spawn将实际编译工作交给独立异步任务：

```rust
pub async fn build_agent_binary(
    &self,
    target_triple: Option<String>,
    listener_id: Option<i64>,
    server_addr: Option<String>,
    agent_token: Option<String>,
    profile: String,
    heartbeat_secs: Option<u64>,
    jitter: Option<u32>,
) -> anyhow::Result<AgentBuildRecord> {
    // Resolve target triple, listener, server_addr...
    let build = self.kernel.storage
        .create_agent_build_record(/* ... */).await?;
    self.kernel.publish_web_event(WebEvent::AgentBuildCreated {
        build: build.clone(),
    });
    let kernel = self.kernel.clone();
    tokio::spawn(async move {
        let result = run_build(/* params */).await;
        // Update build record and notify WebSocket
    });
    Ok(build)
}
```

异步任务内部使用toolchain模块检测本地编译环境，确保指定的target已安装，然后调用cargo build命令。构建过程中通过修改Agent源码中的配置模块注入服务器地址、认证令牌和心跳参数。构建完成后更新数据库记录状态为Succeeded，通过EventBus通知Web控制台。如果构建失败，记录错误信息并通知前端。整个构建过程是非阻塞的，API立即返回build_id，前端通过WebSocket事件跟踪构建进度。

### 5.2.7 代理转发实现

代理转发功能允许操作员通过已上线的Agent建立TCP隧道，访问目标内网中的其他服务。服务端侧的实现涉及ProxyFacade和内核的proxy运行时处理。

操作员发起代理请求时，ProxyFacade通过KernelMessage::Proxy::StartSession消息通知内核。内核创建ProxySessionRecord，同时向Agent下发OpenProxy指令。Agent在目标机器上绑定指定端口后回传ProxyOpened消息。

当有客户端连接到代理端口时，数据流经以下路径：客户端→服务端监听端口→KernelMessage::Proxy::IngressData→Agent→目标主机→Agent回传数据→服务端→客户端。内核通过ProxyStreamRecord追踪每条数据流的元数据，通过mpsc通道在服务端的TCP处理协程和内核消息循环之间传递数据：

```rust
#[derive(Debug)]
pub enum ProxyKernelMessage {
    StartSession {
        agent_id: String,
        proxy_id: String,
        bind_addr: String,
        respond_to: oneshot::Sender<anyhow::Result<ProxySessionSnapshot>>,
    },
    StopSession { proxy_id: String, respond_to: /* ... */ },
    OpenStream {
        proxy_id: String,
        stream_id: String,
        host: String,
        port: u16,
        client_sender: mpsc::UnboundedSender<Option<Vec<u8>>>,
        respond_to: oneshot::Sender<anyhow::Result<()>>,
    },
    ClientData { proxy_id: String, stream_id: String, data: Vec<u8> },
    ClientClosed { proxy_id: String, stream_id: String },
}
```

每条代理流的数据通过base64编码嵌入JSON协议传输，与Agent之间的通信复用同一条TCP连接，无需额外建立网络通道。

## 5.3 客户端核心实现

### 5.3.1 项目结构与组件实现

Web控制台基于Vue 3的组合式API（Composition API）开发，所有代码使用TypeScript编写。项目结构按照功能划分为api/、store/、views/、layouts/和utils/五个目录。api/目录封装了所有与服务端的HTTP通信，每个模块对应一个API域（agent.ts、listener.ts、terminal.ts等）。store/目录包含Pinia状态管理模块，其中events.ts负责WebSocket连接管理和事件分发。views/目录按页面组织，每个页面由index.vue主组件、components/子组件和hooks/组合函数组成。

布局层采用layouts/default/index.vue作为全局框架，包含侧边栏导航（AppSidebar）、顶部标签栏（AppTabs）和通知中心（NotificationCenter）。路由配置在router/index.ts中，使用Vue Router的动态导入实现页面懒加载。

组件间通信遵循Vue 3的标准模式：父子组件通过props和emit传递数据，跨组件状态通过Pinia store共享，实时数据通过WebSocket事件驱动。所有API请求统一经过api/request.ts模块，该模块封装了认证Token的注入和错误处理的通用逻辑。

### 5.3.2 会话管理页面实现

Agent会话管理页面（views/agent/index.vue）是控制台的核心视图之一。页面左侧展示所有已上线和历史上线的Agent列表，右侧是选中Agent的详情面板。列表数据来自两个来源：在线Agent通过WebSocket的Snapshot事件获取实时状态，离线Agent通过HTTP API查询历史记录。

列表采用Ant Design Vue的Table组件渲染，列字段包含主机名、用户名、操作系统、架构、内网IP、监听器名称、心跳间隔、权限级别和在线状态。在线状态通过一个绿色/灰色的圆点指示器直观呈现。每行右侧提供操作菜单（AgentContextMenu），支持打开终端、分发任务、文件操作、代理转发、调整心跳、禁用/启用和断开连接等操作。

点击Agent后进入会话详情页（session.vue），该页面是一个多Tab容器，包含终端、文件管理、截屏等子功能面板。顶部信息栏展示Agent的完整元数据：主机名、用户名、操作系统与架构、权限级别（通过盾牌图标和颜色区分管理员权限与普通用户权限）、在线状态和心跳参数。信息栏的数据由WebSocket事件实时更新，当Agent断开连接时状态指示器自动切换为灰色。

### 5.3.3 交互式终端实现

交互式终端是控制台技术密度最高的组件。其实现分散在四个文件中，各司其职：terminal.vue作为容器组件提供DOM挂载点，useTerminal.ts作为编排层协调初始化流程，useTerminalCore.ts封装xterm.js实例的创建和销毁，useTerminalSocket.ts管理WebSocket连接和消息解析。

terminal.vue的实现非常简洁，DOM部分只有一个挂载终端的div容器，脚本部分将所有逻辑委托给useTerminal组合函数：

```typescript
const { terminalContainer, sessionId, wsConnected } = useTerminal(agentId.value);
```

useTerminal的初始化流程按步骤执行：创建xterm.js实例并挂载到DOM，调用openTerminalSession HTTP API在服务端创建命令会话，连接WebSocket接收实时输出，绑定键盘输入处理。当操作员按下回车键时，输入缓冲区的命令通过submitTerminalCommand API提交，终端显示一行黄色提示文字表明命令已排队。命令执行结果通过WebSocket推送回来后，useTerminalSocket解析payload中的stdout、stderr和exit_code，用xterm.js的write方法渲染到终端界面，同时更新cwd（当前工作目录）用于下一条命令的提示符显示：

```typescript
ws.onmessage = (event) => {
    const payload = JSON.parse(event.data);
    if (payload.type !== 'terminal' || payload.session_id !== sessionId.value) return;
    if (payload.event === 'command' && (payload.state === 'done' || payload.state === 'error')) {
        term?.write('\x1b[1A\x1b[2K'); // Clear "awaiting" line
        if (payload.stdout) {
            term?.write(payload.stdout.replace(/\n/g, '\r\n'));
        }
        if (payload.stderr) {
            term?.write('\x1b[31m' + payload.stderr.replace(/\n/g, '\r\n') + '\x1b[0m');
        }
        if (payload.cwd) cwd.value = payload.cwd;
        onCommandDone();
    }
};
```

终端提示符采用类似Linux shell的格式，用ANSI转义码着色：绿色的hermes标识、蓝色的Agent ID、青色的当前目录，后跟$符号。clear命令被客户端拦截处理，不会发送到服务端。组件销毁时自动关闭WebSocket连接并调用closeTerminalSession API释放服务端资源。

### 5.3.4 文件管理与截屏功能实现

文件管理功能通过session.vue页面中的文件Tab实现，包含目录浏览、文件上传和文件下载三个子功能。目录浏览由路径输入栏和文件列表表格组成。操作员输入路径后点击浏览按钮，前端调用HTTP API向服务端发送browse请求，服务端将请求封装为任务下发给Agent。Agent执行完毕后返回JSON格式的目录条目列表，前端解析后渲染为表格。

文件列表表格区分目录和文件两种条目类型。目录条目显示文件夹图标和名称，点击可进入子目录。文件条目显示文件图标、名称、大小和修改时间，右侧提供下载按钮。表格首行始终显示一个..条目，点击可返回上级目录。

文件上传通过FileOpsModal组件实现，该组件是一个Ant Design Vue的Modal对话框，内含上传和下载两个Tab页。上传流程为：操作员选择本地文件（限制10MB以内），前端通过FileReader读取文件内容并转为base64，然后调用uploadFile API将base64编码的文件内容和目标路径发送到服务端。下载流程更简单：操作员输入远程文件路径，前端调用downloadFile API，服务端将下载任务下发给Agent，Agent读取文件内容后以base64编码回传，前端收到结果后触发浏览器下载。

截屏功能在截屏Tab中实现。操作员点击截屏按钮后，前端调用服务端的screenshot API，服务端将截屏命令下发给Agent。Agent通过系统API捕获屏幕图像，将图片数据以base64编码回传。前端收到结果后在页面中展示截屏图片，并提供下载按钮。

## 5.4 Agent端核心实现

### 5.4.1 通信与心跳机制实现

Agent的通信模块采用同步阻塞模型，不使用任何异步运行时。这一设计选择出于两个考量：减小二进制体积（tokio运行时会显著增加二进制大小），以及降低运行时复杂度。Agent的所有网络操作通过标准库的TcpStream完成，支持可选的TLS加密（通过rustls和tls feature启用）。

Agent的main函数执行以下初始化流程：设置panic hook将崩溃信息写入本地日志文件，加载配置（Config::load），初始化各个Service，然后进入一个无限循环。每次循环执行run_once函数，失败后等待reconnect_secs秒再重试：

```rust
fn main() {
    std::panic::set_hook(Box::new(|panic_info| {
        if let Ok(mut f) = std::fs::OpenOptions::new()
            .create(true).append(true).open("agent_debug.log") {
            let _ = writeln!(f, "[PANIC] {}", panic_info);
        }
    }));

    let cfg = match Config::load() {
        Ok(c) => c,
        Err(_) => std::process::exit(0),
    };

    let (outbox_tx, outbox_rx) = mpsc::channel::<AgentMessage>();
    let network = Arc::new(Mutex::new(NetworkService::new()));
    let heartbeat = Arc::new(Mutex::new(HeartbeatService::new()));
    // ... init other services

    loop {
        if run_once(/* all services */).is_err() {
            // log failure
        }
        kernel.sleep(Duration::from_secs(cfg.reconnect_secs));
    }
}
```

run_once函数完成一次完整的通信周期：建立TCP连接→接收服务端Hello→发送Register消息→进入主循环接收服务端指令→根据心跳间隔发送Heartbeat。主循环中，Agent使用带超时的read_line调用等待服务端消息。收到消息后反序列化为ServerCommand枚举，根据命令类型分发给对应的Service处理。

心跳服务由HeartbeatService实现，管理心跳间隔和抖动计算。每次发送心跳后，通过schedule_from方法计算下次心跳的到期时间。抖动计算使用当前时间戳、进程ID和内部序列号的异或值作为伪随机种子，确保每次心跳间隔不完全相同：

```rust
fn schedule_from(&mut self, now: Instant) {
    let base_ms = self.interval_secs.saturating_mul(1000);
    let max_jitter_ms = base_ms.saturating_mul(self.jitter as u64) / 100;
    let jitter_ms = if max_jitter_ms == 0 { 0 } else {
        self.sequence = self.sequence.wrapping_add(1);
        let seed = SystemTime::now()
            .duration_since(UNIX_EPOCH).unwrap().as_nanos() as u64
            ^ ((std::process::id() as u64) << 16)
            ^ self.sequence;
        seed % (max_jitter_ms + 1)
    };
    self.next_due = now + Duration::from_millis(base_ms.saturating_add(jitter_ms));
}
```

当Agent正在执行交互式命令或代理转发数据时，心跳等待时间会被缩短到100毫秒以内，确保输出数据能够及时回传。这一优化通过检查SessionService和ProxyService的should_poll_fast方法实现。

Agent还维护了一个outbox消息队列。各Service在处理命令时产生的响应消息（如TaskResult、CommandSessionOutputChunk）通过outbox_tx通道发送到队列中，主循环在每次网络读写后通过flush_outbox函数将队列中的消息批量发送到服务端，减少网络往返次数。

### 5.4.2 Shell命令执行实现

命令执行是Agent最核心的能力，分为两种模式：单次任务执行和交互式会话执行。两者都依赖ops.rs中的spawn_shell_process函数作为底层命令启动器。

spawn_shell_process是Agent执行系统命令的底层函数，通过cmd.exe /C调用Windows命令解释器：

```rust
fn spawn_shell_process(command: &str, cwd: Option<&str>) -> std::io::Result<std::process::Child> {
    let mut process = Command::new("cmd");
    let command = windows_shell_command(command);
    process.args(["/C", &command]);
    if let Some(cwd) = cwd {
        process.current_dir(cwd);
    }
    process
        .stdout(Stdio::piped())
        .stderr(Stdio::piped())
        .spawn()
}
```

windows_shell_command函数目前直接返回原始命令字符串，不添加额外的代码页前缀。命令输出的编码转换由下游的decode_output函数处理，这种方式比在每条命令前加chcp 65001更可靠，因为某些Windows环境下chcp命令的行为不完全一致。

单次任务执行由TaskService.dispatch方法处理。该方法首先判断命令类型：如果是upload/download/browse等文件操作，路由到file_ops模块；如果是系统信息查询，路由到sys_ops模块；否则视为shell命令，在新线程中启动子进程执行：

```rust
pub fn dispatch(&mut self, task_id: &str, command: &str, payload: Option<&str>) {
    if super::file_ops::is_file_op(command) {
        // Route to file_ops handler in separate thread
        return;
    }
    if super::sys_ops::is_sys_op(command) {
        // Route to sys_ops handler
        return;
    }
    // Generic shell command
    let child = match spawn_operation(actual_cmd, None, None) { /* ... */ };
    let pid = child.id();
    let result = wait_child(child, timeout_secs);
    // Send TaskResult back via outbox
}
```

命令超时通过wait_child函数实现。当设置了超时时间（timeout_secs > 0）时，子进程在独立线程中等待，主线程通过mpsc::channel的recv_timeout检测是否超时。超时后调用terminate_process函数强制终止子进程，使用taskkill /PID /T /F命令终止整个进程树：

```rust
pub fn terminate_process(pid: u32) -> bool {
    Command::new("taskkill")
        .args(["/PID", &pid.to_string(), "/T", "/F"])
        .stdout(Stdio::null()).stderr(Stdio::null())
        .status().map(|s| s.success()).unwrap_or(false)
}
```

命令取消功能通过RunningTaskHandle实现。每个运行中的任务持有一个AtomicBool标志cancel_requested，当操作员取消任务时，cancel方法设置该标志并尝试终止已启动的子进程。任务线程在子进程启动后立即检查cancel_requested标志，如果已被设置则立即终止进程并返回取消状态。

输出解码针对Windows环境做了特殊处理。中文版Windows默认使用GBK编码（代码页936），子进程输出的原始字节可能不是有效的UTF-8。decode_output函数先尝试UTF-8解码，失败后回退到GBK解码，通过encoding_rs库完成转换：

```rust
pub fn decode_output(raw: &[u8]) -> String {
    if raw.is_empty() { return String::new(); }
    if let Ok(s) = std::str::from_utf8(raw) { return s.to_string(); }
    if cfg!(target_os = "windows") {
        let (decoded, _, _) = encoding_rs::GBK.decode(raw);
        return decoded.to_string();
    }
    String::from_utf8_lossy(raw).to_string()
}
```

交互式会话执行由SessionService负责。与单次任务不同，会话模式需要追踪工作目录的变化。SessionService内部维护一个sessions HashMap，将session_id映射到当前cwd。每次命令执行完毕后，解析cwd_after并更新映射，使得下一条命令能在正确的工作目录下执行。

### 5.4.3 文件操作实现

文件操作模块（file_ops.rs）提供上传、下载和目录浏览三项功能，作为TaskService的分发目标。所有文件操作在独立线程中执行，结果通过outbox通道回传。

上传功能解析JSON格式的payload，提取remote_path和content_base64字段，将base64解码后的二进制数据写入目标路径。如果目标路径的父目录不存在，会自动创建：

```rust
pub fn handle_upload(task_id: &str, payload: &str, sender: &Sender<AgentMessage>) {
    #[derive(serde::Deserialize)]
    struct UploadPayload {
        remote_path: String,
        content_base64: String,
    }
    let parsed: UploadPayload = serde_json::from_str(payload)?;
    let content = STANDARD.decode(&parsed.content_base64)?;
    write_file(&parsed.remote_path, &content)?;
}
```

下载功能读取指定路径的文件内容，通过base64编码后放入TaskResult的output字段回传。目录浏览功能读取指定路径下的所有条目，对每个条目提取名称、大小、修改时间和是否为目录等属性，序列化为JSON后返回。这三个操作都由is_file_op函数统一识别，当TaskService收到的command字段为upload、download或browse时自动路由。

### 5.4.4 代理转发实现

Agent端的代理转发由ProxyService实现，管理从服务端发来的代理指令。ProxyService内部维护两个HashMap：streams以stream_id为键存储到TcpStream的引用，proxy_streams以proxy_id为键存储该代理会话下所有stream_id的列表。

当收到OpenProxy指令时，Agent向服务端回传ProxyOpened确认消息。收到ProxyConnect指令后，Agent尝试建立到目标host:port的TCP连接。连接成功后，将TcpStream通过Arc<Mutex<TcpStream>>包装并插入streams，同时启动一个独立线程持续从该TCP连接读取数据，读取到的数据通过base64编码后以ProxyData消息回传服务端：

```rust
pub fn connect(&mut self, proxy_id: &str, stream_id: &str, host: &str, port: u16) {
    match TcpStream::connect((host, port)) {
        Ok(stream) => {
            let read_stream = stream.try_clone().unwrap();
            let stream = Arc::new(Mutex::new(stream));
            // Register in maps
            let _ = self.sender.send(AgentMessage::ProxyConnectResult {
                proxy_id: proxy_id.to_string(),
                stream_id: stream_id.to_string(),
                success: true,
                detail: None,
            });
            // Spawn reader thread
            std::thread::spawn(move || {
                let mut buf = vec![0u8; 8192];
                loop {
                    match read_stream.read(&mut buf) {
                        Ok(0) => { /* stream closed */ break; }
                        Ok(n) => {
                            let _ = sender.send(AgentMessage::ProxyData {
                                proxy_id: proxy_id.clone(),
                                stream_id: stream_id.clone(),
                                data_base64: STANDARD.encode(&buf[..n]),
                            });
                        }
                        Err(_) => { /* error */ break; }
                    }
                }
            });
        }
        Err(error) => { /* send ProxyConnectResult with failure */ }
    }
}
```

收到ProxyData指令时，Agent从streams中查找对应的TcpStream，将base64解码后的数据写入。close_stream操作将stream从两个HashMap中移除并减少活跃流计数。close_proxy操作清理该代理会话下的所有流。

活跃流计数（active_streams）和flush_hint（flush_hint）是两个AtomicBool/AtomicUsize标志，用于通知主循环当前是否有需要立即回传的数据。当active_streams > 0或flush_hint为true时，主循环的心跳等待时间缩短到100毫秒，保证代理数据能及时转发。数据发送完毕后由主循环调用clear_flush_hint重置标志。

### 5.4.5 异常捕获与进程保护实现

Agent需要在目标主机上长时间稳定运行，异常处理机制是关键。Agent在main函数最开始就设置了自定义panic hook，将panic信息写入同目录下的agent_debug.log文件，而不是让程序直接崩溃退出。Config::load如果失败则静默退出（std::process::exit(0)），不输出任何错误信息，避免暴露Agent的存在。

网络断连处理采用重连机制。run_once函数返回Err时，主循环等待reconnect_secs秒后重新执行整个连接流程。每次重连都会清空outbox队列（clear_outbox），避免将上次会话的残留消息发送到新连接。NetworkService的connect方法设置了30秒的读写超时，read_line方法在超时时返回错误，主循环据此判断是否需要发送心跳或继续等待。

进程信息采集通过sys/native.rs模块实现。该模块根据目标操作系统使用条件编译提供不同的实现。Windows平台通过windows-sys FFI调用Win32 API获取进程权限信息（OpenProcessToken、GetTokenInformation、LookupPrivilegeNameW），区分普通用户、管理员和SYSTEM权限。主机名通过COMPUTERNAME环境变量获取，用户名通过USERNAME环境变量获取，内网IP通过创建一个到8.8.8.8的UDP socket获取本地绑定地址。这些信息在Agent启动时采集一次，随Register消息上报到服务端。

## 5.5 本章小结

本章详细描述了Hermes平台的系统实现过程，涵盖了开发环境配置、服务端微内核运行时、HTTP API路由、WebSocket实时推送、Agent注册与会话管理、交互式终端会话、Agent二进制构建服务、代理转发、Web控制台界面和Agent端通信与执行等核心模块。

服务端采用微内核架构，通过KernelMessage消息调度实现状态集中管理和副作用统一输出。HTTP API和WebSocket作为传输适配层，不包含业务逻辑，所有操作经由Facade→消息→状态变更的路径执行。客户端基于Vue 3和TypeScript实现组件化开发，通过Pinia store管理WebSocket事件驱动的实时状态。Agent以同步阻塞模型运行，针对Windows平台进行优化，利用线程池和原子操作实现并发命令执行，通过panic hook和自动重连机制保障长期稳定运行。

三个子系统通过JSON over TCP协议通信，协议定义在protocol.rs中，采用serde的tag=type枚举序列化，保证了消息类型的自描述性和前向兼容性。整个平台实现了从设计到可运行代码的完整转化，各模块职责清晰，代码可维护性良好。
